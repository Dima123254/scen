# Система построения сценариев угроз ФСТЭК России

Автоматизированная система генерации полного реестра сценариев реализации угроз безопасности информации на основе методологии ФСТЭК России.

## Принцип работы

1. Вы описываете инфраструктуру своей компании в JSON-файле (`input_infrastructure.json`).
2. Скрипт анализирует каждый объект, определяет применимые угрозы, компоненты и уровни нарушителей.
3. Система строит **все возможные** комбинации сценариев (Нарушитель → Угроза → Способ → Объект → Компонент).
4. Для каждого сценария автоматически подбираются негативные последствия и меры защиты.
5. Результат экспортируется в файл `output_scenarios.json`.

## Требования

- Python 3.6+
- Никаких внешних зависимостей (используются только стандартные библиотеки: `json`, `csv`, `io`)

## Запуск

```bash
python scenario_builder.py
```

Скрипт ожидает наличие файла `input_infrastructure.json` в той же директории.

## Формат входного файла (input_infrastructure.json)

Файл должен содержать корневой объект с массивом `objects`. Каждый объект описывает элемент инфраструктуры вашей компании.

### Структура объекта

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| `id` | string | ✅ | Уникальный идентификатор объекта в вашей инфраструктуре (например, "WS_Accounting_01") |
| `type` | string | ✅ | Тип объекта из справочника ФСТЭК (О.1, О.2, О.10 и т.д.) |
| `components` | array | ❌ | Список компонентов (К.x.x.x), присутствующих в объекте. Если не указано — используются **все возможные** для данного типа |
| `attackers` | array | ❌ | Список уровней нарушителей (В.1, В.2, В.3, В.4), которых нужно учитывать. Если не указано — перебираются **все уровни** |

### Пример файла

```json
{
  "objects": [
    {
      "id": "WS_BUH_01",
      "name": "Рабочее место бухгалтера",
      "type": "О.1",
      "components": [
        "К.1.2.1",
        "К.1.5.11",
        "К.1.5.15"
      ],
      "attackers": ["В.1", "В.2"]
    },
    {
      "id": "SRV_DB_01",
      "name": "Сервер баз данных",
      "type": "О.2",
      "components": [],
      "attackers": []
    },
    {
      "id": "MOB_DIR_01",
      "name": "Телефон директора",
      "type": "О.10",
      "attackers": ["В.2", "В.3", "В.4"]
    }
  ]
}
```

### Пояснения к примеру

1. **WS_BUH_01 (О.1 - АРМ)**:
   - Указаны конкретные компоненты: ОС, Офисное ПО, Браузер.
   - Учитываются нарушители уровня В.1 и В.2.
   - Система построит сценарии только для указанных компонентов и уровней.

2. **SRV_DB_01 (О.2 - Сервер)**:
   - Компоненты не указаны (`[]` или отсутствие поля) — система автоматически подставит **все возможные** компоненты для серверов (ОС, СУБД, Прошивка и т.д.).
   - Нарушители не указаны — будут перебраны **все уровни** (В.1, В.2, В.3, В.4).
   - Это даст максимально полный перечень сценариев для сервера.

3. **MOB_DIR_01 (О.10 - Мобильное устройство)**:
   - Компоненты не указаны — будут использованы все возможные для мобильных устройств.
   - Фильтр по нарушителям: только В.2, В.3, В.4 (базовый уровень В.1 исключен).

| ID | Наименование | Описание |
|----|--------------|----------|

## Логика валидации

Система автоматически проверяет транзитивность связей:
1. **Нарушитель → Способ**: Уровень нарушителя должен быть >= требуемого уровня способа.
2. **Способ → Угроза**: Способ должен реализовывать данную угрозу.
3. **Угроза → Объект**: Угроза должна быть применима к типу объекта (по матрице ФСТЭК).
4. **Компонент → Объект**: Компонент должен входить в состав объекта.
5. **Угроза → Последствие**: Последствие должно быть привязано к угрозе.
6. **Мера → Способ**: Мера должна парировать данный способ реализации.

Если какая-либо связь отсутствует, сценарий не генерируется.

---

## Примечания

- **Расчет критичности**: В текущей версии поле `criticality` всегда равно `0.0`. Методика расчета будет добавлена отдельно в будущих версиях.
- **Производительность**: Для одного объекта с полным набором компонентов и всеми уровнями нарушителей может быть сгенерировано от 500 до 2000 сценариев. Рекомендуется указывать конкретные компоненты для ускорения обработки.
- **Обновление баз**: Справочники ФСТЭК встроены в скрипт. Для обновления версий необходимо редактировать константы `DB_*` в начале файла `scenario_builder.py`.
